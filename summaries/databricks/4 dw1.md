### 1. The Core Theory: The Four Pillars of a Data Warehouse
Slide 7 introduces the "Golden Words" of Bill Inmon, the father of the Data Warehouse. He states a DW must be four things. This is foundational Big Data theory:

1.  **Subject-Oriented (Slide 8):** Operational (OLTP) systems are built around *apps* (e.g., a loan processing app). A Data Warehouse is built around *business entities* (e.g., viewing all data related to a "Customer," regardless of what app it came from).
2.  **Time-Variant (Slide 9):** OLTP only cares about *right now*. A Data Warehouse keeps a historical record (snapshots). If you want to compare Q1 2020 to Q1 2023, the data must be time-variant.
3.  **Integrated (Slide 10):** Taking messy data from 10 different systems, resolving naming conflicts (e.g., "Cust_ID" vs "ClientNo"), and merging it into a single, clean format.
4.  **Non-Volatile (Slide 11):** In a traditional DW, once data is written, it is read-only. You don't delete historical data; you just add new records.

### 2. "Run the Business" vs. "Optimize the Business" (Slides 12-13)
This is the best summary of OLTP vs. DW:
*   **OLTP runs the business.** If it goes down, customers can't buy things. It handles gigabytes of data and thousands of simultaneous users.
*   **The DW optimizes the business.** If it goes down, the CEO can't read their daily dashboard, but customers can still buy things. It handles terabytes/petabytes of data and massive batch queries.

### 🧠 The "Databricks Philosophy" Connection

How does Databricks take these 1990s Data Warehouse concepts and revolutionize them for the 2020s?

**1. "Databricks SQL" – The DW Killer:**
Historically, building a Data Warehouse required expensive, proprietary hardware/software (like Oracle, Teradata, or Snowflake). Databricks introduced a feature called **Databricks SQL**. It provides a user interface and an incredibly fast computing engine (called **Photon**) that allows business analysts to write standard SQL queries just like they would in a traditional DW—but it runs directly on the cheap, massive Data Lake!

**2. Delta Lake "Time Travel" (Solving Time-Variant & Non-Volatile):**
Slides 9 and 11 mention that a DW must keep historical snapshots and be non-volatile. In the old days, Data Engineers had to write complex ETL code to save daily copies of the data. 
*   **The Databricks Solution:** Databricks uses Delta Lake, which has a feature called **Time Travel**. Because Delta Lake keeps a running log of every single transaction, it is *natively* time-variant. In Databricks, if you want to see what a table looked like exactly 30 days ago, you literally just type: 
    `SELECT * FROM my_table TIMESTAMP AS OF '2023-09-01'` 
    You get the history automatically without having to build complex snapshot pipelines!

**3. Integration via Unity Catalog:**
Slide 10 talks about integration. Databricks handles this via **Unity Catalog**, which allows you to define a single source of truth for all your structured (tables) and unstructured (images/files) data, integrating everything securely in one place.
