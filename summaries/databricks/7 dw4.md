While extraction sounds simple (just copying data from point A to point B), how you extract data makes or breaks a Big Data pipeline. Here is the theoretical breakdown and how Databricks has completely automated this process.

### 1. The Core Theory: The Extraction Phase
When you pull data from a source system (like a MySQL database or an API), you have to make two major architectural decisions:

**Where does it go? (Slide 6)**
*   Data isn't transformed immediately as it's pulled. It first lands in a **Landing Area** (or Staging Area). This is a safe, temporary storage zone where raw data is dumped exactly as it looks in the source system, just in case something goes wrong later in the pipeline.

**How much do we pull? (Slides 8-9)**
*   **Full Extraction:** You pull *everything*. If the source table has 10 million rows, you pull 10 million rows. You only do this for an "Initial Load" (the very first time you set up the pipeline) or a massive system migration.
*   **Incremental Extraction (Delta Extraction):** You only pull the data that is *new* or has *changed* since the last time you ran the extraction. You usually define a "Change Window" (e.g., "Give me all records created between yesterday at midnight and today at midnight"). 

### 🚀 The "Databricks Philosophy" Connection

Handling Incremental (Delta) Extractions is historically one of the most frustrating parts of Data Engineering. You had to write complex code to remember exactly which timestamps or IDs you already processed, and if a job failed, you had to manually reset those timestamps.

Databricks looked at this problem and essentially solved it with a few built-in tools:

**1. The Landing Area = The Bronze Layer:**
In Databricks (using the Medallion Architecture we discussed earlier), the "Landing Area" is your **Bronze Layer**. You extract the data from the source and dump it straight into Bronze tables as raw JSON, CSV, or Parquet. No cleaning, no filtering.

**2. Databricks Auto Loader (Solving Incremental Extraction):**
If you are extracting files from cloud storage (like AWS S3 or Azure Data Lake), Databricks provides a magical feature called **Auto Loader**. 
*   Instead of you writing code to figure out which files are new and which ones you've already processed, Auto Loader does it automatically. You just point it at a folder, and it keeps a hidden ledger. Every time you run it, it only extracts the brand-new files. It completely automates Incremental Extraction!

**3. Spark Structured Streaming:**
If you need real-time Incremental Extraction (e.g., reading a constant feed of Twitter data), Databricks uses Apache Spark Structured Streaming. It treats data like a continuous river rather than a static pond, incrementally extracting and processing data 24/7 without you ever having to define "Start" or "End" times.

*(Note: In the transcript, they use the term "Delta extraction" to mean incremental changes. In Databricks, the word "Delta" is everywhere—Delta Lake, Delta Tables—but the core concept is the same: dealing only with the differences/changes rather than the whole dataset).*
