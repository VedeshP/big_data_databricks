### 1. The Core Theory: Understanding Facts (Slides 6-7)
Fact tables hold metrics, but not all metrics can be treated the same way when analyzing data.
*   **Additive:** You can add them across all dimensions. (e.g., *Total Sales.* You can sum up Total Sales by day, by store, or by product, and the math always works).
*   **Semi-Additive:** You can add them across *some* dimensions, but not time. (e.g., *Bank Account Balance.* You cannot add Monday's balance ($100) and Tuesday's balance ($100) and say your total balance is $200. You have to take a snapshot at the end of the month).
*   **Non-Additive:** You cannot add them at all. (e.g., *Profit Margin %.* If store A has a 10% margin and Store B has a 10% margin, your total margin isn't 20%. You must calculate an average).

### 2. The Core Theory: Slowly Changing Dimensions (SCDs) (Slides 9-12)
This is a guaranteed interview question for any Big Data role. When the descriptive data in your Dimension tables changes (e.g., a customer moves to a new city, or a product changes its name), how do you update the database?

*   **SCD Type 1 (Overwrite):** You simply delete the old city and overwrite it with the new city. 
    *   *Pros:* Easy to do, keeps tables small. 
    *   *Cons:* You lose history. You can never query where the customer *used* to live.
*   **SCD Type 2 (Add a New Row):** (Not explicitly detailed in the slide, but crucial to know). You keep the old row, mark it as "Inactive", and insert a brand new row with the new city marked as "Active". 
    *   *Pros:* Perfect historical tracking. 
    *   *Cons:* Tables get very large.
*   **SCD Type 3 (Add a New Column):** You keep the current row, but you add columns like "Original Location" and "Current Location".
    *   *Pros:* Keeps tables short. 
    *   *Cons:* You only keep partial history (if they move 3 times, you lose data).

### 3. Other Dimension Types (Slides 13-16)
*   **Degenerate Dimensions:** A dimension that sits *inside* the Fact table because it doesn't need its own table (e.g., a simple Invoice Number).
*   **Junk Dimensions:** A single table that holds a bunch of random flags (Yes/No, M/F) so you don't have to create 50 tiny tables for every little boolean value.
*   **Conformed Dimensions:** A dimension that is identically built and shared across multiple Fact tables (e.g., A "Date" dimension used by both the Sales team and the HR team).

### 🚀 The "Databricks Philosophy" Connection

How does Databricks make handling Facts and SCDs easier than traditional systems?

**1. Managing SCDs with Delta Lake (`MERGE INTO`):**
In traditional Data Lakes (using flat CSV or Parquet files), doing an **SCD Type 1 (Overwrite)** or **SCD Type 2 (Update/Insert)** was virtually impossible. You couldn't just update one row in a 50GB file; you had to rewrite the whole file.

Databricks fixed this with **Delta Lake**. Delta Lake supports a SQL command called `MERGE INTO` (also known as an "Upsert"). 
*   With one simple line of code, Databricks can check if a Customer ID already exists. If it exists, it can perform an SCD Type 1 or Type 2 update. If it doesn't exist, it inserts it as a new row. This revolutionized Big Data engineering.

**2. Databricks Delta Live Tables (DLT) for SCD Type 2:**
Databricks released a framework called **Delta Live Tables (DLT)**. If you want to track perfect history (SCD Type 2), you don't even have to write complex SQL code anymore. DLT has a built-in function called `APPLY CHANGES INTO`. You just tell it "track changes on the Customer City column," and Databricks automatically manages the historical rows, active flags, and timestamps behind the scenes!
